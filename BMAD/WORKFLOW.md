# BMAD Project Workflow

This document outlines the workflow processes and procedures for the BMAD project.

## Project Overview

BMAD is a project with integrated configuration management and documentation systems designed to streamline development and project management.

## Workflow Structure

### 1. Documentation Organization
- `docs/` - Main documentation directory
- `docs/sprint-artifacts/` - Sprint planning and tracking documents
- `docs/technical/` - Technical documentation and specifications
- `docs/architecture/` - Architecture decisions and system design
- `docs/workflows/` - Process documentation and guidelines

### 2. Configuration Management
- `.bmad/` - Core BMAD configuration
- `.agentvibes/` - Agent configuration and settings  
- `.claude/` - Claude AI integration settings

## Sprint Management Process

### Sprint Planning
1. Create sprint goals in `docs/sprint-artifacts/sprint-XX-planning.md`
2. Define user stories and acceptance criteria
3. Estimate story points and assign tasks
4. Document sprint backlog

### Daily Workflow
1. Check current sprint status
2. Update task progress
3. Document blockers and issues
4. Communicate with team members

### Sprint Review & Retrospective
1. Review completed work against sprint goals
2. Document lessons learned
3. Identify improvement opportunities
4. Plan next sprint adjustments

## Task Management

### Task Categories
- **Feature Development** - New functionality implementation
- **Bug Fixes** - Issue resolution and debugging
- **Technical Debt** - Code improvements and refactoring
- **Documentation** - Updates to project documentation
- **DevOps** - Infrastructure and deployment improvements

### Task Workflow States
1. **Backlog** - Identified but not yet started
2. **In Progress** - Currently being worked on
3. **Code Review** - Ready for peer review
4. **Testing** - Under quality assurance
5. **Done** - Completed and deployed

## Development Guidelines

### Code Quality Standards
- Follow established coding conventions
- Write comprehensive tests
- Document complex logic
- Perform peer code reviews
- Maintain clean commit history

### Definition of Done
- [ ] Code is written and tested
- [ ] Unit tests pass
- [ ] Code review completed
- [ ] Documentation updated
- [ ] No critical issues identified
- [ ] Acceptance criteria met

## Communication Guidelines

### Status Updates
- Daily standup participation

- Retrospective feedback

### Documentation Standards
- Use clear, concise language
- Include relevant examples
- Maintain version control
- Regular updates and reviews
